# 华为 GaussDB Windows 版 Python 驱动需求分析

## 1. 背景

当前项目目标是提供一个支持华为 GaussDB 的 Python 驱动包，重点面向 Windows 环境，并支持 SQLAlchemy 架构。首期目标数据库版本为 GaussDB 轻量化集中式 505.1。

GaussDB 轻量化集中式部署通常面向资源受限、部署简单、应用侧改造成本低的场景。应用开发侧希望像使用 PostgreSQL 或其他 SQLAlchemy 数据库后端一样，通过标准 Python 生态完成连接、连接池、事务、ORM、SQL 执行和基础反射。

本项目当前采用轻量方案：底层复用华为官方 `gaussdb` Python DB-API 包，上层提供 SQLAlchemy 方言适配。这样可以避免重新实现数据库通信协议，优先解决应用侧接入问题。

## 2. 建设目标

### 2.1 总体目标

建设一个可在 Windows 环境安装和使用的 GaussDB Python 驱动项目，支持 SQLAlchemy 2.x 接入，满足 GaussDB 轻量化集中式 505.1 的基础应用开发需求。

### 2.2 阶段目标

第一阶段：最小可用版本

- 支持 pip 安装
- 支持 Windows 环境安装说明
- 支持 SQLAlchemy 方言注册
- 支持 `gaussdb://` 和 `gaussdb+gaussdb://` 连接串
- 支持基础连接参数转换
- 支持 SQLAlchemy 连接池、事务和 SQL 执行的基础链路
- 提供中文 README、示例代码和基础单元测试

第二阶段：真实环境验证版本

- 接入真实 GaussDB 轻量化集中式 505.1 测试环境
- 验证 Windows 下客户端库加载、连接、认证和 SSL 配置
- 验证常用 DDL、DML、事务、批量插入、连接池回收
- 补充 SQLAlchemy ORM 场景测试
- 梳理 GaussDB 与 PostgreSQL 行为差异

第三阶段：增强适配版本

- 适配 GaussDB 特有数据类型
- 适配系统表和元数据反射差异
- 适配主键、自增、序列、默认值、索引、约束反射
- 增加 SQLAlchemy 方言兼容性测试套件
- 支持离线安装包、Windows 部署脚本和版本发布流程

## 3. 用户和使用场景

### 3.1 目标用户

- 使用 Python 开发业务系统的应用开发人员
- 需要将 PostgreSQL 或其他数据库应用迁移到 GaussDB 的团队
- 使用 SQLAlchemy ORM、SQLAlchemy Core、Alembic 的后端开发人员
- 在 Windows 开发机或 Windows Server 上连接 GaussDB 的交付团队

### 3.2 典型场景

场景一：SQLAlchemy Core 查询

应用通过 `create_engine()` 创建 GaussDB 连接，使用 `text()` 执行 SQL，并通过连接池复用连接。

场景二：SQLAlchemy ORM 应用

业务系统定义 ORM Model，使用 Session 完成增删改查、事务提交和回滚。

场景三：Windows 开发环境连接 GaussDB

开发人员在 Windows 上安装 Python、GaussDB 客户端库和本驱动包，连接公司内部 GaussDB 轻量化集中式实例。

场景四：迁移 PostgreSQL 应用

已有 Python + SQLAlchemy + PostgreSQL 应用希望低成本迁移到 GaussDB，连接串从 `postgresql+psycopg://` 调整为 `gaussdb+gaussdb://`，并逐步处理数据库行为差异。

## 4. 功能需求

- 支持 query string 透传，例如 `sslmode`、`application_name`

## 7. 风险分析

### 7.1 GaussDB 与 PostgreSQL 行为差异

SQLAlchemy 的 PostgreSQL 基类可以覆盖大量基础能力，但 GaussDB 可能在系统表、数据类型、函数、DDL、序列、自增、错误码等方面存在差异。

应对措施：

- 建立真实 GaussDB 505.1 测试环境
- 逐项验证 SQLAlchemy Core、ORM 和反射能力
- 将差异固化为方言适配代码和测试用例

### 7.2 Windows 客户端库加载风险

`gaussdb` 包运行时需要可用的 GaussDB/libpq 客户端实现。Windows 上常见问题是 DLL、PATH、VC Runtime 或客户端版本不匹配。

应对措施：

- 编写 Windows 安装检查脚本
- 提供 PATH 配置示例
- 梳理常见错误和排查步骤
- 后续考虑制作离线依赖包或安装脚本

### 7.3 SQLAlchemy 版本演进风险

SQLAlchemy 2.x 内部方言接口可能演进，长期需要跟进兼容性。

应对措施：

- 在依赖中限制兼容范围
- 建立 CI 测试矩阵
- 定期验证 SQLAlchemy 新版本

### 7.4 当前缺少真实数据库验证

当前测试主要是无数据库单元测试，没有验证真实连接和 SQL 执行。

应对措施：

- 增加真实 GaussDB 集成测试
- 增加 Windows 环境验证记录
- 增加典型业务 SQL 回归用例
